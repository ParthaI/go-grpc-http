2. Entry Point — Where GC Begins

GC is not a single function — it's a cooperative dance between three actors:

The Mutator
Your goroutines. They allocate memory and may be forced to assist when allocation is fast.
Background Workers
Dedicated goroutines launched per P. They run gcBgMarkWorker and mark objects.
Sysmon
The runtime monitor thread. Forces GC if none has run in 2 minutes.

All three paths eventually call gcStart() in mgc.go — the unified entry point.

runtime/malloc.go — allocation may trigger GC
func mallocgc(size uintptr, typ *_type, needzero bool) unsafe.Pointer {
    // ... (fast path: pop from mcache) ...

    // Check if we should start a GC cycle
    if shouldhelpgc {
        if t := (gcTrigger{kind: gcTriggerHeap}); t.test() {
            gcStart(t)  // <── GC begins here
        }
    }
    return x
}
runtime/mgc.go — forced GC via runtime.GC()
// runtime.GC() — user-called forced collection
func GC() {
    n := work.cycles.Load()
    gcWaitOnMark(n)
    gcStart(gcTrigger{kind: gcTriggerCycle, n: n + 1})
    gcWaitOnMark(n + 1)
    // Then trigger sweep to completion
    ...
}

3. The GC State Machine (gcphase)

The runtime tracks a global GC phase. Every goroutine, write barrier, and allocation checks this variable.

runtime/mgc.go
const (
    _GCoff             = iota // GC not running; sweeping in background
    _GCmark                   // GC marking roots and workbufs: allocate black
    _GCmarktermination        // GC mark termination: allocate black, P's help GC
)

var gcphase uint32   // global, atomic-accessed
var writeBarrier struct {
    enabled bool     // compiler emits write barrier calls?
    pad     [3]byte
    needed  bool
    alignme uint64
}

// setGCPhase is called whenever the phase transitions
func setGCPhase(x uint32) {
    atomic.Store(&gcphase, x)
    writeBarrier.needed = x == _GCmark || x == _GCmarktermination
    writeBarrier.enabled = writeBarrier.needed
}
Key insight: The write barrier is ONLY active during _GCmark and _GCmarktermination. During _GCoff, pointer writes are plain stores — zero overhead.

4. gcStart — Kicking Off a Cycle

gcStart() is a dense function (~100 lines). Here's a simplified view:

runtime/mgc.go (simplified)
func gcStart(trigger gcTrigger) {
    // 1. Lock out other GC triggers
    semacquire(&work.startSema)

    // 2. Re-check trigger condition (another goroutine might have started GC)
    if !trigger.test() {
        semrelease(&work.startSema)
        return
    }

    // 3. Complete any pending sweep work before starting a new cycle
    for trigger.kind == gcTriggerHeap && atomic.Load(&mheap_.sweepgen) == ... {
        sweep.nbgsweep++
        Gosched()
    }

    // 4. Start the STW phase — "sweep termination"
    systemstack(stopTheWorldWithSema)  // ── STW BEGINS

    // 5. Finish remaining sweep work
    systemstack(func() {
        finishsweep_m()
    })

    // 6. Prepare for marking: clear work buffers, scan counters
    work.cycles.Add(1)
    gcController.startCycle(...)
    work.heap0 = gcController.heapLive.Load()

    // 7. Enable write barrier — compiler emits checks for writeBarrier.enabled
    setGCPhase(_GCmark)

    // 8. Start background mark workers (one per P)
    gcBgMarkStartWorkers()

    // 9. Mark roots: globals, finalizers, bss, data
    gcMarkRootPrepare()

    // 10. Mark all currently running Gs as having "dirty" stacks
    gcMarkTinyAllocs()

    // 11. Resume the world — marking proceeds concurrently
    systemstack(startTheWorldWithSema)  // ── STW ENDS
}

After gcStart returns, the world is running again but in _GCmark phase. Background workers begin consuming the work queue.

5. STW — How the World Actually Stops

Stop-the-world is misleadingly named. The runtime doesn't halt execution mid-instruction — it waits for every goroutine to reach a safe point.

runtime/proc.go (simplified)
func stopTheWorldWithSema() {
    gp := getg()

    // 1. Set a global flag requesting all Ps to yield
    sched.stopwait = gomaxprocs
    atomic.Store(&sched.gcwaiting, 1)

    // 2. Preempt all running goroutines
    preemptall()

    // 3. Stop each P when it reaches a safe point
    for _, pp := range allp {
        s := pp.status
        if s == _Prunning && atomic.Cas(&pp.status, s, _Pgcstop) {
            sched.stopwait--
        }
    }

    // 4. Wait until all Ps have stopped
    if sched.stopwait != 0 {
        for {
            // Re-preempt if any P still running
            if notetsleep(&sched.stopnote, 100*1000) {
                break
            }
            preemptall()
        }
    }
}

How Safe Points Work

Since Go 1.14, goroutines are asynchronously preemptible. The runtime sends a SIGURG signal to threads running Go code. The signal handler checks gp.preempt and parks the goroutine.

Before Go 1.14: preemption only happened at function calls. A tight loop with no calls could block GC indefinitely. Signal-based preemption fixed this.

6. Concurrent Mark — The Work Queue CONCURRENT

Marking is driven by a producer-consumer work queue of grey objects. Workers dequeue grey objects, scan their pointers, enqueue newly-found grey objects, then mark them black.

runtime/mgcmark.go — the mark worker loop
func gcBgMarkWorker() {
    gp := getg()
    for {
        // Sleep until woken by the pacer
        gopark(...)

        // Claim mark work based on worker mode
        switch pp.gcMarkWorkerMode {
        case gcMarkWorkerDedicatedMode:
            gcDrain(&pp.gcw, gcDrainUntilPreempt|gcDrainFlushBgCredit)
        case gcMarkWorkerFractionalMode:
            gcDrain(&pp.gcw, gcDrainFractional|gcDrainUntilPreempt|gcDrainFlushBgCredit)
        case gcMarkWorkerIdleMode:
            gcDrain(&pp.gcw, gcDrainIdle|gcDrainUntilPreempt|gcDrainFlushBgCredit)
        }

        // Check if all work is done
        if gcMarkDone() {
            gcMarkTermination()  // <── Triggers STW mark termination
        }
    }
}

gcDrain — The Actual Marking Loop

runtime/mgcmark.go (simplified)
func gcDrain(gcw *gcWork, flags gcDrainFlags) {
    // 1. First, scan any global/stack roots that need processing
    for work.markrootNext < work.markrootJobs {
        job := atomic.Xadd(&work.markrootNext, 1) - 1
        markroot(gcw, job)
    }

    // 2. Drain the grey work queue
    for !(preemptible && gp.preempt) {
        // Take next grey object from workbuf
        b := gcw.tryGet()
        if b == 0 {
            break  // No work left
        }
        scanobject(b, gcw)  // Scan its pointers, greying children
    }
}

scanobject — Greying Children

runtime/mgcmark.go (simplified)
func scanobject(b uintptr, gcw *gcWork) {
    // Look up type metadata to know which fields are pointers
    s := spanOfUnchecked(b)
    hbits := heapBitsForAddr(b)
    n := s.elemsize

    // Walk each potential pointer slot
    for i := uintptr(0); i < n; i += goarch.PtrSize {
        if hbits.isPointer() {
            obj := *(*uintptr)(unsafe.Pointer(b + i))
            if obj != 0 && inheap(obj) {
                if obj, span, objIndex := findObject(obj, b, i); obj != 0 {
                    greyobject(obj, b, i, span, gcw, objIndex)
                }
            }
        }
        hbits = hbits.next()
    }
}

func greyobject(obj, base, off uintptr, span *mspan, gcw *gcWork, objIndex uintptr) {
    mbits := span.markBitsForIndex(objIndex)
    if mbits.isMarked() {
        return  // Already grey or black — skip
    }
    mbits.setMarked()  // Mark it
    gcw.putFast(obj)   // Enqueue for scanning (grey)
}
Note on "color": Go doesn't literally store three colors. It uses a single mark bit per object. WHITE = unmarked. GREY = marked AND in a work queue. BLACK = marked AND not in any queue. This is clever and space-efficient.

7. The Write Barrier in Code

The Go compiler inserts a call to gcWriteBarrier around every pointer store when writeBarrier.enabled == true.

runtime/mbarrier.go — the hybrid barrier
// gcWriteBarrier is the compiler-inserted entry point.
// src/runtime/asm_amd64.s contains the hand-written assembly.
// It ultimately calls:

func wbBufFlush1(pp *p) {
    // Drain the per-P write barrier buffer into the grey queue
    buf := &pp.wbBuf
    n := buf.next - buf.end
    ptrs := buf.buf[:n]

    for _, ptr := range ptrs {
        obj, span, objIndex := findObject(ptr, 0, 0)
        if obj == 0 {
            continue
        }
        mbits := span.markBitsForIndex(objIndex)
        if mbits.isMarked() {
            continue
        }
        mbits.setMarked()  // Shade it
        gcw.putFast(obj)   // Enqueue
    }
    buf.reset()
}

The Hybrid Barrier Logic (Conceptual)

What the compiler emits for: slot = ptr
// Pseudocode for "slot = ptr" when write barrier is on:
if writeBarrier.enabled {
    // Dijkstra insertion: shade the new value
    shade(ptr)
    // Yuasa deletion: shade the old value
    shade(*slot)
}
*slot = ptr
Why both insertion AND deletion? The insertion barrier alone requires re-scanning goroutine stacks at mark termination (expensive STW). The deletion barrier lets stacks stay BLACK permanently after initial scan. The hybrid gets the best of both — zero stack re-scan, low overhead.

The Per-P Write Barrier Buffer

Calling shade on every write would be slow. Instead, Go uses a per-P ring buffer (wbBuf) that batches pointers. When the buffer fills, it's flushed into the global work queue.

runtime/mwbbuf.go
type wbBuf struct {
    next uintptr       // Pointer into buf[]
    end  uintptr       // End of buf[]
    buf  [wbBufEntries]uintptr  // ~256 pointer slots
}

8. Mark Termination STW

When gcMarkDone() determines all workers are idle AND all work buffers are empty, it transitions into mark termination.

runtime/mgc.go (simplified)
func gcMarkTermination() {
    // 1. Stop the world (again, briefly)
    systemstack(stopTheWorldWithSema)

    // 2. Flush all mcaches — forces final allocations to be accounted
    systemstack(func() {
        forEachP(func(pp *p) {
            pp.mcache.prepareForSweep()
        })
    })

    // 3. Drain any last work added by write barriers during the STW transition
    gcMark(startTime)

    // 4. Transition to off-phase; disable write barrier
    setGCPhase(_GCoff)

    // 5. Begin the sweep phase
    gcSweep(work.mode)

    // 6. Update GC controller state for pacing
    gcController.endCycle(...)

    // 7. Resume the world
    systemstack(startTheWorldWithSema)
}
Typical duration: The entire mark termination STW is usually 50–500µs. Most of it is flushing mcaches and bookkeeping, NOT marking (that's essentially done).

9. Sweep — Lazy Reclamation CONCURRENT

After the mark is finished, every unmarked (WHITE) object is garbage. Sweep reclaims them — but lazily, per-span, on demand.

runtime/mgcsweep.go (simplified)
func sweepone() uintptr {
    // Find the next span that needs sweeping
    s := mheap_.central[spc].mcentral.partialUnswept(sg).pop()
    if s == nil {
        return ^uintptr(0)  // No work
    }

    // Walk the span's mark bits
    nfree := 0
    for i := uintptr(0); i < s.nelems; i++ {
        if !s.markBitsForIndex(i).isMarked() {
            // Object was NOT marked → it's garbage → free slot
            s.freeindex = i
            nfree++
        }
    }

    // Return span to free pool or keep for reuse
    if s.allocCount == 0 {
        mheap_.freeSpan(s)
    }
    return uintptr(nfree)
}

Two Sweep Drivers

  • Background sweeper — a dedicated goroutine (bgsweep) slowly processes spans.
  • On-demand sweep — when a goroutine allocates and its mcache needs refilling, it first sweeps the span it's about to use.
Why lazy? Eager sweep would waste CPU sweeping memory that nobody needs yet. Lazy sweep amortizes the cost across allocations that actually happen.

10. The Pacer Controller

The pacer is the "brain" of the GC. It answers: when should we start the next cycle, and how much CPU should workers get?

runtime/mgcpacer.go (simplified)
type gcControllerState struct {
    // Target heap size for next GC, in bytes
    heapGoal atomic.Uint64

    // Live heap bytes at end of last mark
    heapMarked uint64

    // Bytes allocated since start of GC
    heapLive atomic.Uint64

    // PI controller state
    consMark         float64  // Estimated mark cost (CPU/byte)
    scannableStackSize atomic.Uint64
    gcPercent        atomic.Int32  // GOGC value

    // Heap trigger point (gcStart fires when heapLive reaches this)
    trigger atomic.Uint64
}

func (c *gcControllerState) endCycle(now int64, procs int, userForced bool) {
    // PI controller: adjust consMark based on actual vs predicted work
    currentConsMark := (float64(c.heapScanWork) * assistRatio) / ...

    // Update estimate with proportional + integral terms
    oldConsMark := c.consMark
    c.consMark = oldConsMark + gcControllerKp*(currentConsMark-oldConsMark) + ...
}

func (c *gcControllerState) commit() {
    // Compute heap goal from GOGC
    goal := c.heapMarked + c.heapMarked*uint64(c.gcPercent.Load())/100

    // Trigger earlier than goal so marking finishes in time
    runway := c.consMark * float64(c.scannableStackSize.Load()) / (1 - gcGoalUtilization)
    trigger := goal - uint64(runway)

    c.trigger.Store(trigger)
    c.heapGoal.Store(goal)
}
PI controller in plain English:
  • P (Proportional): if this cycle overshot the target, trigger earlier next time by a proportional amount.
  • I (Integral): if overshoots keep happening, accumulate the bias and push trigger even earlier.
Same math airplanes and thermostats use. It's remarkable that a GC trigger is a control-theory problem.

11. Allocation Path — mallocgc

Every new(), make(), and struct literal that escapes to the heap goes through mallocgc. This function is also where mutator assist is enforced.

runtime/malloc.go (heavily simplified)
func mallocgc(size uintptr, typ *_type, needzero bool) unsafe.Pointer {
    mp := acquirem()

    // 1. If GC is running and we're allocating fast, help with marking
    assistG := (*g)(nil)
    if gcBlackenEnabled != 0 {
        assistG = getg()
        assistG.gcAssistBytes -= int64(size)
        if assistG.gcAssistBytes < 0 {
            gcAssistAlloc(assistG)   // <── MUTATOR ASSIST
        }
    }

    // 2. Fast path: allocate from the per-P mcache
    c := getMCache(mp)
    var span *mspan
    var x unsafe.Pointer

    if size <= maxSmallSize {
        // Small object path (most allocations)
        sizeclass := size_to_class[divRoundUp(size, smallSizeDiv)]
        span = c.alloc[sizeclass]
        v := nextFreeFast(span)
        if v == 0 {
            v, span, shouldhelpgc = c.nextFree(sizeclass)
        }
        x = unsafe.Pointer(v)
    } else {
        // Large object path (> 32 KB) — directly from mheap
        span = c.allocLarge(size, noscan)
        x = unsafe.Pointer(span.base())
    }

    // 3. If this allocation might have crossed the GC trigger, start a cycle
    if shouldhelpgc {
        if t := (gcTrigger{kind: gcTriggerHeap}); t.test() {
            gcStart(t)
        }
    }

    // 4. If we're in mark phase, new objects are born BLACK (already scanned)
    //    This is called "allocate-black" — prevents new allocations from
    //    being freed before they can be used.
    return x
}

gcAssistAlloc — Paying for Your Allocation

runtime/mgcmark.go
func gcAssistAlloc(gp *g) {
    for gp.gcAssistBytes < 0 {
        // How much scan work does this goroutine owe?
        scanWork := int64(-gp.gcAssistBytes) * int64(gcController.assistWorkPerByte.Load())

        // Do the work — scan some grey objects yourself
        workDone := gcDrainN(&getg().m.p.ptr().gcw, scanWork)

        // Credit back
        gp.gcAssistBytes += int64(workDone) / int64(gcController.assistBytesPerWork.Load())
    }
}
Real-world impact: If your service is spending a lot of time in runtime.gcAssistAlloc (visible in pprof CPU profiles), you're allocating faster than the GC can keep up. Options: reduce allocation rate, increase GOGC, or use sync.Pool.
