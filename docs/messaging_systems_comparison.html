🚀 RabbitMQ vs MQTT vs NATS

Production-ready Go implementations — TLS, reconnection, confirms, DLQs, idempotency

🐰 RabbitMQ

The Reliable Enterprise Broker (AMQP)

Producer → Exchange → Queue → Consumer
  • Routing: direct, topic, fanout, headers
  • Guaranteed delivery + ACKs
  • Dead-letter queues
  • Transactional publishing
  • Complex enterprise workflows

📡 MQTT

The Lightweight IoT Protocol

Device → Broker → Subscribers
topic: sensors/room1/temp
  • Ultra-lightweight (2-byte minimum header)
  • QoS 0, 1, 2 delivery modes
  • Retained messages + LWT
  • Works on unreliable networks
  • Perfect for battery-powered devices

⚡ NATS

The Cloud-Native Speed Demon

Publisher → NATS Server → Subscribers
subject: orders.created
  • Sub-millisecond latency
  • Millions of msgs/sec
  • Pub/Sub, Request/Reply, Queue groups
  • JetStream for persistence
  • Built-in clustering

🛡️ Production Hardening Checklist — What Every Deployment Must Have

🔐 Security

  • TLS 1.2+ on all connections
  • Mutual TLS (mTLS) for MQTT devices
  • JWT / NKEYS for NATS auth
  • RBAC / ACLs per topic/queue
  • Secrets from Vault / KMS — never hardcoded
  • Network isolation (VPC, security groups)

🔄 Reliability

  • Auto-reconnect with exponential backoff
  • Publisher confirms (RabbitMQ) / Acks (JetStream)
  • Dead-letter queues for poison messages
  • Max retry limits + retry backoff
  • Idempotent consumers (dedup keys)
  • Graceful shutdown with drain

📊 Observability

  • Prometheus metrics (publish rate, ack latency)
  • OpenTelemetry trace context propagation
  • Consumer lag monitoring + alerts
  • Queue depth + DLQ size dashboards
  • Structured logging with correlation IDs
  • Broker health checks in readiness probes

⚖️ Scale & HA

  • Clustered brokers (RabbitMQ quorum queues)
  • NATS superclusters across regions
  • MQTT broker bridging / clustering
  • Replication factor ≥ 3 for JetStream
  • Horizontal consumer scaling (queue groups)
  • Prefetch / flow control tuning

💾 Data Integrity

  • Persistent storage with fsync
  • Message TTLs to prevent unbounded growth
  • Schema validation (Protobuf / Avro / JSON Schema)
  • Deduplication windows (JetStream Duplicates)
  • Backup + disaster recovery plan
  • Stream compaction / retention policies

🚨 Operational Readiness

  • Runbooks for broker failure scenarios
  • Chaos testing (kill brokers, network partitions)
  • Load testing matching production traffic
  • Capacity planning + autoscaling rules
  • On-call rotation + escalation paths
  • SLOs for publish success & e2e latency

🧭 When to Use What? — Detailed Decision Guide

🐰 Pick RabbitMQ if you need...
  • Complex message routing (direct, topic, fanout, headers exchanges)
  • Strong delivery guarantees with manual ACKs & re-delivery
  • Dead-letter queues for failed message handling
  • Priority queues for urgent tasks
  • Transactional publishing (atomic multi-message ops)
  • Plugin ecosystem (Shovel, Federation, Management UI)
  • Battle-tested stability for enterprise workflows
  • Support for multiple protocols (AMQP, STOMP, MQTT via plugin)
💼 Ideal for: Banking transactions, order processing, job queues, ETL pipelines, workflow orchestration, microservice integration
Ultra-high throughput (>500K msg/s) or IoT on low-power devices
📡 Pick MQTT if you need...
  • Minimal bandwidth & battery usage (2-byte header)
  • Support for millions of concurrent device connections
  • Three QoS levels (0 fire-forget, 1 at-least-once, 2 exactly-once)
  • Last Will & Testament for device offline detection
  • Retained messages (new subscribers get last known state)
  • Hierarchical topic filtering with wildcards (+, #)
  • Works on flaky, high-latency, or unstable networks
  • Standardized protocol (OASIS/ISO) with huge broker ecosystem
🌐 Ideal for: IoT sensor networks, smart home devices, connected cars, industrial telemetry, mobile push notifications, asset tracking
Complex routing logic or high-throughput backend messaging
⚡ Pick NATS if you need...
  • Sub-millisecond latency (<1ms p99)
  • Extreme throughput (millions of messages/second)
  • Simple pub/sub with subject-based addressing
  • Request/Reply pattern built into the protocol
  • Queue groups for automatic load balancing
  • Native clustering & supercluster for global scale
  • Tiny binary (~20MB) — easy to deploy anywhere
  • Cloud-native design (CNCF graduated project)
  • Zero external dependencies & minimal configuration
🚀 Ideal for: Microservice mesh, real-time analytics, gaming backends, edge computing, service discovery, telemetry streaming
Complex routing rules or when you need AMQP compatibility
💾 Pick NATS JetStream if you need...
  • NATS-level speed PLUS message persistence
  • Exactly-once delivery semantics
  • Stream replay & time-travel consumption
  • Durable consumers that survive restarts
  • Kafka-like streaming with simpler operations
  • Built-in deduplication & flow control
  • Key-Value store & Object store capabilities
  • Multi-tenancy with accounts & permissions
📊 Ideal for: Event sourcing, audit logs, cloud-native event streaming, replayable pipelines, CQRS architectures
Small-scale apps where RabbitMQ's simplicity is enough
⚠️ Common Pitfalls to Avoid
  • Don't use MQTT for backend-to-backend — it's built for devices
  • Don't use RabbitMQ as a log store — use Kafka or JetStream
  • Don't use plain NATS when you need durability — use JetStream
  • Don't mix protocols without a clear gateway strategy
  • Don't over-engineer — sometimes a simple Redis pub/sub works
  • Don't ignore back-pressure & consumer lag monitoring
  • Don't forget to secure brokers (TLS, auth, ACLs)
🎯 Quick Decision Matrix
  • IoT devices? → MQTT
  • Enterprise workflows? → RabbitMQ
  • Microservices at scale? → NATS
  • Event streaming? → NATS JetStream / Kafka
  • Need exactly-once? → RabbitMQ or JetStream
  • Low-power sensors? → MQTT (QoS 0)
  • Request/Reply RPC? → NATS
  • Complex routing rules? → RabbitMQ topic exchange
